### Question 1001: What is the loop variable scope change in Go 1.22?

**Answer:**
Before Go 1.22, loop variables were shared across iterations, causing bugs in closures (capturing the last value).
In Go 1.22, the loop variable is **per-iteration**.
```go
// Go 1.22+
for i := 0; i < 3; i++ {
    go func() {
        fmt.Println(i) // Now prints 0, 1, 2 correctly (order undefined of course)
    }()
}
```

### Explanation
Go 1.22 changed loop variable scope from being shared across all iterations to being per-iteration. This fixes the common closure bug where all goroutines captured the same loop variable value, eliminating the need for manual variable copying.

### How to Explain in Interview (Spoken style format)
**Interviewer:** What is the loop variable scope change in Go 1.22?
**Your Response:** "Go 1.22 fixed a major source of bugs by changing loop variable scope. Before, loop variables were shared across all iterations, so if I created goroutines in a loop, they'd all capture the same final value. Now, each iteration gets its own variable, so closures work as expected. This eliminates the need for workarounds like `i := i` to copy the variable. The change makes Go more intuitive and prevents a whole class of common bugs. It's particularly important for concurrent code with goroutines. This was a breaking change but worth it for the bug prevention. The key is that now each loop iteration has its own variable scope, making closures behave predictably."

### Question 1021: How do you implement the "Or-Done" channel pattern in Go?

                return
            case v, ok := <-c:

                select {

                }
            }
        }
    }()
    return valStream
}

### Explanation
The Or-Done channel pattern wraps channel reads with a context/done channel check to avoid blocking forever. It uses a select statement to check for cancellation while reading from a channel and forwarding values.

### Question 1022: What is a "Tee-Channel" and how do you implement it?

**Answer:**

```go
func tee(in <-chan int) (<-chan int, <-chan int) {
    out1 := make(chan int)

        defer close(out1); defer close(out2)
        for val := range in {
            out1 <- val
            out2 <- val
        }
    }()
    return out1, out2
}
```

### Explanation
The Tee-Channel pattern splits one input channel into two identical output channels, similar to the Unix tee command. It reads from input and sends each value to both output channels concurrently.
